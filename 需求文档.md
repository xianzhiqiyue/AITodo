# AI 任务调度中心 - 需求文档

---

## 1. 产品需求文档 (PRD)

### 1.1 项目目标

构建一个**"AI 优先"的任务调度中心**。它不仅是存储列表的地方，更是 AI 的"外挂大脑"，用于：

- 记录指令
- 拆解目标
- 跟踪执行进度

### 1.2 核心用户故事

#### 模糊输入转结构化
用户说"下周给老王发个报告"，AI 自动提取时间、对象、任务类型并入库。

#### 任务递归拆解
用户说"写一个 ToDo 模块"，AI 调用 Skill 将其拆分为：

1. 数据库设计
2. API 编写
3. Docker 配置

三个子任务。

#### 上下文关联
AI 在执行其他任务（如编写代码）时，能主动查询是否有相关的待办项未关闭。

### 1.3 功能需求

| 功能 | 描述 |
|------|------|
| 基础 CRUD | 支持通过 API 进行任务的增删改查 |
| 状态流转 | `todo` → `in_progress` → `done`（支持 `blocked` 挂起状态，详见 2.2 状态枚举） |
| AI 增强元数据 (Metadata) | 允许 AI 存储非结构化信息，如：对话摘要、参考链接、执行日志 |
| 任务层级 | 支持父子任务关系（Parent-Child Tasks） |
| 智能检索 | 支持基于语义或标签的过滤 |

### 1.4 非功能需求

| 需求 | 说明 |
|------|------|
| 机器友好 | 必须具备标准的 OpenAPI (Swagger) 文档 |
| 安全性 | 基于 API Key 的访问控制 |
| 轻量化 | Docker 一键部署，内存占用极低 |

## 2. 系统设计文档 (SDD)

### 2.1 技术栈

| 组件 | 选型 | 原因 |
|------|------|------|
| 语言 | Python 3.11+ (FastAPI) | 对异步支持极好，自动生成 OpenAPI 规范 |
| 数据库 | PostgreSQL | 使用 JSONB 存储元数据 |
| 部署 | Docker & Docker Compose | 容器化部署 |
| AI 协议 | MCP + Nanobot/OpenClaw | 双协议支持：MCP 兼容 Cursor/Claude，REST 兼容 Nanobot/OpenClaw（详见 2.7） |

### 2.2 数据模型 (Schema)

#### Tasks 表结构

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | UUID | Primary Key | 任务唯一标识 |
| `title` | VARCHAR(255) | Not Null | 任务简述 |
| `description` | TEXT | Nullable | 详细内容/Markdown |
| `status` | VARCHAR(20) | Default 'todo' | 状态枚举见下方 |
| `priority` | INTEGER | Default 3 | 优先级 1-5，1 = 最高，5 = 最低 |
| `due_at` | TIMESTAMP | Nullable | 截止时间 |
| `parent_id` | UUID | FK → Tasks.id, Nullable | 父任务 ID |
| `tags` | TEXT[] | Default '{}' | 标签数组，支持多标签过滤 |
| `meta_data` | JSONB | Default '{}' | 存储 AI 提取的上下文 |
| `embedding` | vector(1536) | Nullable | 语义向量（pgvector 扩展） |
| `created_at` | TIMESTAMP | Default NOW() | 创建时间 |
| `updated_at` | TIMESTAMP | Default NOW() | 最后更新时间，每次修改自动刷新 |

#### 状态枚举

| 状态值 | 含义 | 备注 |
|--------|------|------|
| `todo` | 待办 | 默认初始状态 |
| `in_progress` | 进行中 | 已开始执行 |
| `done` | 已完成 | 任务结束 |
| `blocked` | 挂起/阻塞 | 等待外部条件，可从任何状态转入 |

合法状态流转：`todo` → `in_progress` → `done`，任意状态均可转入 `blocked`，`blocked` 可恢复至 `todo` 或 `in_progress`。

#### 父子任务规则

| 规则 | 说明 |
|------|------|
| 嵌套层级 | 最大 5 层 |
| 删除约束 | 存在子任务时禁止删除父任务，需先删除或移除子任务 |
| 完成约束 | 父任务标记为 `done` 前，所有子任务必须为 `done` |
| 状态联动 | 父任务进入 `blocked` 时，不自动改变子任务状态 |

### 2.3 AI Skill (Tool) 接口设计

AI 工具的核心在于 Description 的编写。以下是提供给 AI 的关键 Tool 定义（同时适用于 MCP 和 REST 两种协议，详见 2.7）：

#### `upsert_task`

**描述**：当用户提到新的任务、修改任务或改变进度时调用。不提供 `id` 时为新建，提供 `id` 时为更新。

**参数**：

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `id` | string (UUID) | ❌ | 任务 ID，提供时为更新已有任务，不提供时新建 |
| `title` | string | ✅(新建) | 任务标题，更新时可选 |
| `description` | string | ❌ | 任务详细描述，支持 Markdown |
| `status` | string | ❌ | 状态值：`todo` / `in_progress` / `done` / `blocked` |
| `priority` | integer | ❌ | 优先级 1-5，1 = 最高 |
| `due_at` | string (ISO 8601) | ❌ | 截止时间，如 `2025-12-31T23:59:59Z` |
| `parent_id` | string (UUID) | ❌ | 父任务 ID，用于建立层级关系 |
| `tags` | array of string | ❌ | 标签列表，如 `["backend", "urgent"]` |
| `meta_data` | object | ❌ | 自由格式 JSON，存储 AI 上下文 |
| `thinking_process` | string | ❌ | 写入 meta_data.thinking，AI 对该任务的理解 |

**返回**：创建/更新后的完整任务对象。

#### `get_task_context`

**描述**：获取当前相关的所有待办项，用于辅助 AI 决策。支持按状态、标签、语义进行过滤。

**参数**：

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `status_filter` | string | ❌ | 默认 `open`。可选值：`open`(= todo + in_progress)、`todo`、`in_progress`、`done`、`blocked`、`all` |
| `top_n` | integer | ❌ | 返回条数，默认 20，最大 100 |
| `offset` | integer | ❌ | 分页偏移量，默认 0 |
| `tags` | array of string | ❌ | 按标签过滤，返回包含任一指定标签的任务 |
| `query` | string | ❌ | 语义搜索关键词，使用向量相似度匹配（需启用 pgvector） |
| `parent_id` | string (UUID) | ❌ | 仅返回指定父任务的子任务 |

**返回**：`{ "tasks": [...], "total": number, "offset": number }`

#### `delete_task`

**描述**：删除指定任务。默认情况下，存在子任务时会拒绝删除；设置 `cascade=true` 可同时删除所有子任务。

**参数**：

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `task_id` | string (UUID) | ✅ | 要删除的任务 ID |
| `cascade` | boolean | ❌ | 默认 `false`。为 `true` 时级联删除所有子任务 |

**返回**：`{ "deleted_count": number }`

#### `decompose_task`

**描述**：将一个任务拆解为多个子任务。AI 识别到用户需要拆解目标时调用，自动创建子任务并关联 `parent_id`。

**参数**：

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `task_id` | string (UUID) | ✅ | 要拆解的父任务 ID |
| `sub_tasks` | array of object | ✅ | 子任务列表，每项包含 `title`(必填)、`description`(可选)、`priority`(可选)、`due_at`(可选) |

**返回**：`{ "parent_task": {...}, "sub_tasks": [...] }`

### 2.4 安全与认证

#### API Key 认证

所有 REST API 请求必须携带 API Key，通过 HTTP Header 传递：

```
Authorization: Bearer <API_KEY>
```

MCP 协议通过 stdio 通信，不经过 HTTP，因此不需要 API Key 认证（由宿主进程负责安全隔离）。

#### 速率限制

| 限制项 | 值 | 说明 |
|--------|------|------|
| 请求频率 | 100 req/min | 按 API Key 维度限流 |
| 响应策略 | HTTP 429 | 超限时返回 `Too Many Requests` |
| 限流算法 | 滑动窗口 | 使用内存计数器实现 |

#### 统一错误响应格式

所有 API 错误均返回以下 JSON 结构：

```json
{
  "error": {
    "code": "TASK_NOT_FOUND",
    "message": "Task with id 'xxx' does not exist."
  }
}
```

常见错误码：

| 错误码 | HTTP 状态码 | 说明 |
|--------|-------------|------|
| `VALIDATION_ERROR` | 400 | 参数校验失败 |
| `UNAUTHORIZED` | 401 | API Key 无效或缺失 |
| `TASK_NOT_FOUND` | 404 | 任务不存在 |
| `HAS_CHILDREN` | 409 | 存在子任务，无法删除 |
| `PARENT_NOT_DONE` | 409 | 子任务未全部完成，父任务无法标记为 done |
| `MAX_DEPTH_EXCEEDED` | 422 | 超过最大嵌套层级 (5 层) |
| `RATE_LIMITED` | 429 | 请求频率超限 |
| `INTERNAL_ERROR` | 500 | 服务内部错误 |

### 2.6 语义检索方案

#### 技术选型

| 组件 | 选型 | 说明 |
|------|------|------|
| 向量存储 | PostgreSQL `pgvector` 扩展 | 无需独立向量数据库，与业务数据同库 |
| 嵌入模型 | OpenAI `text-embedding-3-small` | 1536 维，性价比高；支持替换为任何兼容 OpenAI API 格式的模型 |
| 相似度算法 | 余弦距离 (`<=>`) | pgvector 内置支持 |

#### 工作流程

1. **写入时**：创建或更新任务时，将 `title + description` 拼接后调用嵌入模型，生成 1536 维向量存入 `embedding` 字段
2. **查询时**：`get_task_context` 接收 `query` 参数，先将 query 转为向量，再用 `<=>` 算子在 `embedding` 字段上做近似最近邻搜索
3. **索引**：在 `embedding` 字段上创建 IVFFlat 或 HNSW 索引以加速搜索

#### SQL 示例

```sql
-- 启用扩展
CREATE EXTENSION IF NOT EXISTS vector;

-- 创建索引
CREATE INDEX idx_tasks_embedding ON tasks
  USING hnsw (embedding vector_cosine_ops);

-- 语义搜索查询
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM tasks
WHERE status IN ('todo', 'in_progress')
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

#### 降级策略

当嵌入服务不可用时（如 API Key 未配置或服务超时），自动降级为基于 `title`/`tags` 的关键词搜索（PostgreSQL `ILIKE` + `@>` 数组包含），不影响核心功能。

### 2.7 双协议支持（MCP + REST）

#### 架构设计

核心业务逻辑封装在 `TaskService` 中，通过适配器模式同时暴露两种接口：

| 协议 | 入口文件 | 传输方式 | 适用场景 |
|------|----------|----------|----------|
| MCP (Model Context Protocol) | `mcp_server.py` | stdio (标准输入输出) | Cursor、Claude Desktop 等本地 AI 工具 |
| REST API (FastAPI) | `main.py` | HTTP | Nanobot、OpenClaw、浏览器、第三方集成 |

#### MCP Server 定义

MCP Server 将注册以下 Tools（与 REST API 的 Tool 一一对应）：

| MCP Tool 名称 | 对应 REST 端点 | 说明 |
|----------------|----------------|------|
| `upsert_task` | `POST /api/v1/tasks` | 创建或更新任务 |
| `get_task_context` | `GET /api/v1/tasks` | 查询任务列表 |
| `delete_task` | `DELETE /api/v1/tasks/{id}` | 删除任务 |
| `decompose_task` | `POST /api/v1/tasks/{id}/decompose` | 拆解任务 |

#### REST API 路由设计

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/v1/tasks` | 创建任务 |
| `PUT` | `/api/v1/tasks/{id}` | 更新任务 |
| `GET` | `/api/v1/tasks` | 查询任务列表（支持过滤/分页/语义搜索） |
| `GET` | `/api/v1/tasks/{id}` | 获取单个任务详情（含子任务） |
| `DELETE` | `/api/v1/tasks/{id}` | 删除任务 |
| `POST` | `/api/v1/tasks/{id}/decompose` | 拆解任务为子任务 |
| `GET` | `/health` | 健康检查 |

#### Cursor MCP 配置示例

```json
{
  "mcpServers": {
    "ai-todo": {
      "command": "python",
      "args": ["mcp_server.py"],
      "cwd": "/path/to/todoServer"
    }
  }
}
```

## 3. 针对 AI 工具的优化建议

### 3.1 System Prompt 引导

在 AI 配置中加入以下指引（适用于 MCP 和 Nanobot/OpenClaw 两种接入方式）：

> "你拥有一个长效记忆工具 todo_service。当你意识到用户在布置任务、或者你完成了一个阶段性目标时，必须同步更新到该工具中。在开始新任务前，先调用 get_task_context 检查是否有相关未完成项。"

### 3.2 错误处理逻辑

当 API 返回错误时，AI 应根据错误码自动处理：

| 错误码 | AI 处理策略 |
|--------|-------------|
| `VALIDATION_ERROR` | 自动修正参数格式后重试 |
| `HAS_CHILDREN` | 提示用户确认是否级联删除 |
| `PARENT_NOT_DONE` | 列出未完成的子任务，逐个处理 |
| `MAX_DEPTH_EXCEEDED` | 将超出层级的子任务平铺到合法层级 |
| `RATE_LIMITED` | 等待后重试 |

### 3.3 定时巡检 (Cron AI)

可以在 Docker 中跑一个简单的脚本，每天定时唤醒一次 AI，让它查询 `get_task_context` 并生成"今日任务简报"。
