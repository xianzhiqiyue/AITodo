# AITodo 规划与开发计划

## 1. 需求文档

### 1.1 项目定位

AITodo 是一个面向 AI Agent 和人类协作的任务中枢。它负责将自然语言输入转换成结构化任务，维护任务之间的层级、依赖、进展与上下文，并向 REST 客户端或 MCP Agent 提供可执行的任务视图。

### 1.2 产品目标

1. 让用户或 Agent 能快速记录任务。
2. 让系统能表达拆解、依赖、阻塞和执行上下文。
3. 让 AI 能基于任务状态和关系做出下一步决策。
4. 让系统同时适配 REST 与 MCP 两种接入方式。

### 1.3 核心用户

1. 借助 AI 进行项目管理的个人开发者。
2. 需要稳定任务上下文的 Agent 开发者。
3. 希望整合待办、执行记录和提醒的小团队。

### 1.4 核心场景

1. 自然语言录入任务，并由系统生成结构化草稿。
2. 对复杂目标进行任务拆解。
3. 查询当前可启动任务、阻塞任务和即将到期任务。
4. 记录任务执行过程中的评论、日志与外部链接。
5. 通过依赖关系表达真实执行顺序。

### 1.5 功能需求

1. 任务管理：CRUD、状态流转、层级、搜索、删除约束。
2. 自然语言解析：草稿生成、候选方案、置信度评分、确认入库。
3. 任务拆解：显式拆解与后续 AI 建议拆解。
4. 任务依赖：依赖 CRUD、环检测、可启动判断。
5. 工作台视图：`today`、`overdue`、`blocked`、`ready_to_start`、`recently_updated`。
6. 评论与执行日志：时间线、系统事件、失败记录、参考链接。
7. 提醒与巡检：逾期、即将到期、长期阻塞任务聚合。
8. 外部接入：REST API、MCP tools、统一错误响应。

### 1.6 非功能需求

1. API Key 鉴权与基础限流。
2. PostgreSQL 生产可用，SQLite 测试可用。
3. 清晰分层，测试覆盖主要业务规则。
4. 结构化日志、健康检查、后续可扩展 metrics。

## 2. 架构设计

### 2.1 总体架构

当前采用单体后端架构，但内部按职责分层：

1. 接口层：FastAPI routes、MCP tools。
2. 应用编排层：TaskIntakeService 等跨服务编排。
3. 领域服务层：TaskService、TaskParsingService。
4. 持久化层：SQLAlchemy models + Alembic。
5. 外部适配层：Embedding API、LLM parsing API。

### 2.2 核心数据流

1. 自然语言入库：文本 -> 解析草稿 -> 置信度门槛 -> 选择候选/覆盖 -> 创建任务。
2. 执行决策：任务查询 -> 状态/依赖/截止时间综合判断 -> 返回工作台视图。
3. 状态变更：输入校验 -> 领域规则校验 -> 提交数据库 -> 返回更新对象。

### 2.3 部署架构

1. FastAPI + Uvicorn。
2. PostgreSQL + pgvector。
3. MCP stdio server。
4. Dockerfile + docker-compose 部署。

### 2.4 演进方向

1. 后续可拆 background worker 处理提醒与扫描任务。
2. LLM 解析增强后，可独立拆成 AI 服务模块。
3. 前端接入后可新增工作台聚合 API 或 BFF。

## 3. 模块设计

### 3.1 Task Core

职责：

1. 任务 CRUD。
2. 状态机校验。
3. 父子层级与防环。
4. 搜索与删除约束。

### 3.2 Task Parsing

职责：

1. 自然语言抽取任务草稿。
2. 候选草稿生成。
3. 时间、标签、优先级解析。
4. 置信度评分。

### 3.3 Task Intake

职责：

1. 解析与入库编排。
2. 草稿选择。
3. override 合并。
4. 低置信度拦截。

### 3.4 Task Dependency

职责：

1. 管理任务依赖关系。
2. 检测依赖环。
3. 计算任务是否可启动。
4. 提供 `ready_to_start` 查询。

### 3.5 Comment / Timeline

职责：

1. 评论与执行日志。
2. 失败记录。
3. 系统事件。
4. 时间线聚合。

### 3.6 Workspace / Alerts

职责：

1. 今日任务视图。
2. 逾期、阻塞、可启动任务聚合。
3. 后续对接通知与后台任务。

## 4. 开发计划

### 阶段 1：任务依赖与可启动视图

1. 新增任务依赖模型与迁移。
2. 实现依赖 CRUD。
3. 实现依赖环检测。
4. 增加 `ready_to_start` 视图接口与测试。

### 阶段 2：任务过程沉淀

1. 新增评论与时间线。
2. 自动写入系统事件。
3. 支持进展、失败、链接记录。

### 阶段 3：工作台与告警

1. 新增 `today`、`overdue`、`blocked`、`recently_updated`。
2. 增加逾期与长期阻塞巡检接口。

### 阶段 4：AI 拆解增强

1. 增加拆解建议接口。
2. 支持确认后批量创建子任务。
3. 尝试生成推荐依赖顺序。

### 阶段 5：后台提醒与通知

1. 后台扫描任务。
2. 通知输出接口。
3. 后续扩展 webhook / 邮件 / IM。

1. 任务依赖关系。
2. `ready_to_start` 查询。
3. 相关约束、接口和测试。
