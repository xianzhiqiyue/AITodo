# 下一阶段模块设计与开发文档

## 下一阶段目标

当前主干能力已经完成，下一阶段不再以“补核心 CRUD/解析链路”为主，而是进入“工程化稳定性 + 产品化增强”阶段。目标是三件事：

1. 让系统持续可交付：迁移、测试、发布、回滚都稳定。
2. 让系统可运营：能看见错误、性能、风险任务和系统状态。
3. 让系统更主动：不仅存任务，还能提醒、汇总、辅助决策。

下面给出一份完整的下一阶段模块设计文档和开发文档。

## 下一阶段模块设计文档

### 1. 阶段定义

#### 阶段名称
`v1.1 工程化与可运营化阶段`

#### 阶段目标
在现有任务中枢基础上，补齐工程质量、运行可观测性、自动化校验和主动提醒能力，让系统从“功能完整”进入“可长期演进”。

#### 设计原则
- 不破坏现有 API 主干。
- 优先补基础设施能力，不盲目扩新业务面。
- 所有新增模块都应有明确边界，避免把逻辑塞回 `TaskService`。
- 保持 PostgreSQL 为生产主路径，SQLite 为开发/测试兼容路径。

### 2. 模块总览

下一阶段建议新增或增强 5 个模块：

1. CI / Delivery 模块
2. Migration Guard 模块
3. Observability 模块
4. Notification / Alert Delivery 模块
5. Workspace Aggregation 模块增强

### 3. 模块设计

### 3.1 CI / Delivery 模块

#### 目标
让每次变更都自动验证“代码可运行、数据库可升级、测试可通过”。

#### 解决的问题
- 改了 schema 但迁移不可用。
- 本地能跑，CI/新环境跑不起来。
- 测试遗漏导致回归。

#### 模块职责
- 自动执行测试。
- 自动执行 Alembic 升级检查。
- 自动执行基础 lint / import 校验。
- 后续支持构建镜像和发布前检查。

#### 输入
- Git 提交或 Pull Request。

#### 输出
- CI 成功 / 失败结果。
- 明确失败阶段：依赖安装、迁移、测试、打包。

#### 设计内容
建议增加：
- `pytest -q`
- `alembic upgrade head`
- `alembic downgrade base`
- 再次 `alembic upgrade head`
- 可选 `python -m compileall .`

#### 边界
- 不负责业务提醒。
- 不负责线上部署。
- 只负责“代码变更是否健康”。

### 3.2 Migration Guard 模块

#### 目标
把数据库迁移从“脚本集合”提升成“受约束的演进机制”。

#### 解决的问题
- 迁移脚本方言不兼容。
- 多次升级/降级不一致。
- schema 与 models 漂移。

#### 模块职责
- 统一管理迁移策略。
- 校验 SQLite / PostgreSQL 兼容性。
- 为未来新增表、索引、约束提供规范。

#### 建议规则
1. 所有迁移必须显式处理 PostgreSQL / SQLite 差异。
2. 所有新索引都要说明 SQLite fallback。
3. 禁止依赖“仅生产库可执行”的裸 SQL，除非有 fallback。
4. 每个迁移都要可 downgrade。

#### 后续可扩展点
- 自动 schema drift 检查。
- 迁移命名规范检查。
- migration smoke test 工具化。

### 3.3 Observability 模块

#### 目标
让系统运行状态可见，便于排障和运营。

#### 解决的问题
- 错了不知道哪里错。
- 慢接口无感知。
- 请求链路无法追踪。
- 任务风险无法量化。

#### 模块职责
- 请求级追踪标识。
- 结构化日志增强。
- 慢请求记录。
- 错误聚合信息输出。
- 健康检查增强。

#### 设计内容

##### 3.3.1 Request ID
每个请求生成 `request_id`，写入：
- 响应头
- 应用日志
- 错误日志

##### 3.3.2 Slow Request Logging
当请求耗时超过阈值，例如 `500ms` 或 `1000ms`，记录：
- path
- method
- duration
- status_code
- request_id

##### 3.3.3 Error Logging
未捕获异常时记录：
- exception type
- message
- stack trace
- request_id
- path

##### 3.3.4 Health Check 增强
当前 `/health` 可扩展为：
- `status`
- `database`
- `version`
- `migration`
- `parsing_service`
- `embedding_service`

##### 3.3.5 Metrics 预留
后续可接 Prometheus，先定义指标口径：
- 请求总数
- 错误总数
- 慢请求数
- 创建任务数
- 解析成功率
- 低置信度拒绝数

#### 边界
- 先做日志与基础健康检查。
- 暂不引入复杂监控栈。

### 3.4 Notification / Alert Delivery 模块

#### 目标
让提醒从“可查询”升级到“可主动触达”。

#### 解决的问题
- 现在只能查 alerts，系统不会主动告诉你。
- 风险任务不能定时推送。

#### 模块职责
- 调用现有告警扫描结果。
- 生成通知事件。
- 推送到外部通道。

#### 第一阶段设计
先做抽象层，不直接耦合具体渠道。

#### 核心组件

##### AlertDeliveryService
职责：
- 获取 `list_alerts()` / `scan_reminders()` 结果
- 根据规则过滤
- 组装通知 payload
- 调用 provider 发送

##### NotificationProvider 接口
定义方法：
- `send(message: str, payload: dict) -> None`

#### Provider 预留实现
- WebhookProvider
- DingTalkProvider
- EmailProvider

#### 通知规则
初版建议仅覆盖：
- overdue
- due_soon
- blocked too long

#### 去重策略
避免重复轰炸：
- 同一任务同一告警类型在 N 小时内只发一次
- 记录最后发送时间到表或缓存

#### 后续数据结构建议
`notification_deliveries`
- `id`
- `task_id`
- `reason`
- `channel`
- `status`
- `sent_at`
- `meta_data`

### 3.5 Workspace Aggregation 模块增强

#### 目标
把已有工作台接口从“零散列表”升级成“统一视图中心”。

#### 解决的问题
- 现在接口分散，调用方需要自己拼。
- Agent 或前端难以一次性获得总览。

#### 模块职责
- 聚合 today / overdue / blocked / ready-to-start / recently-updated
- 输出统一 dashboard 结果
- 为未来 UI 和 Agent 提供单入口

#### 建议新增接口
`GET /api/v1/workspace/dashboard`

#### 返回结构建议
- `today`
- `overdue`
- `blocked`
- `ready_to_start`
- `recently_updated`
- `alerts_summary`

#### 设计原则
- 单接口总览，内部复用现有 service。
- 不复制现有逻辑。
- 保持 top_n 可配置。

#### 价值
- 前端接入更简单。
- Agent 获取上下文更直接。
- 可作为首页或每日摘要基础。

## 下一阶段开发文档

### 1. 开发范围

#### 范围内
- CI 自动检查
- Alembic 迁移守护
- 日志与健康检查增强
- 主动提醒发送能力抽象
- 工作台聚合总览接口

#### 暂不包含
- 前端页面开发
- 真正复杂的调度引擎
- 多租户
- 权限系统升级
- 大规模消息队列架构

### 2. 里程碑计划

### 里程碑 1：CI 与迁移守护

#### 目标
保证每次变更都能自动验证基础健康。

#### 开发项
1. 增加 CI 配置
2. 增加测试步骤
3. 增加 Alembic upgrade/downgrade 检查
4. 补充 README 中的 CI / migration 说明

#### 交付物
- `.github/workflows/ci.yml`
- 文档更新
- 稳定通过的 CI pipeline

#### 验收标准
- 新 PR 自动触发检查
- 失败能明确定位阶段
- 迁移坏掉时 CI 必须失败

### 里程碑 2：Observability 增强

#### 目标
让线上问题可见、可定位。

#### 开发项
1. 增加 `request_id` 中间件
2. 增加慢请求日志
3. 增强异常日志
4. 增强 `/health`

#### 涉及文件建议
- `app/api/middleware.py`
- `main.py`
- `app/logging.py`
- `app/api/routes.py`
- `app/schemas.py`

#### 验收标准
- 所有 API 请求日志带 `request_id`
- 未捕获异常日志可定位请求
- `/health` 输出更完整状态信息

### 里程碑 3：Workspace Dashboard

#### 目标
提供统一工作台聚合接口。

#### 开发项
1. 设计 dashboard schema
2. 新增 service 编排层
3. 新增 REST 接口
4. MCP 是否补同名工具可同步考虑

#### 建议新增对象
- `WorkspaceDashboardResponse`
- `WorkspaceSummaryItem`

#### 验收标准
- 一个接口返回多类任务总览
- 逻辑复用现有 service，不重复实现
- 补 API 测试

### 里程碑 4：Notification 抽象与主动提醒

#### 目标
让提醒具备主动触达能力。

#### 开发项
1. 定义 `NotificationProvider`
2. 实现 `WebhookProvider`
3. 实现 `AlertDeliveryService`
4. 新增手动触发接口
5. 可选新增 delivery 记录表

#### 建议接口
- `POST /api/v1/notifications/dispatch-alerts`
- 后续后台 job 调这个服务

#### 验收标准
- 能把 alerts 结果组装并发送到 webhook
- 失败可记录日志
- 重复发送有基本抑制策略

### 里程碑 5：文档与运维补齐

#### 目标
降低交接和维护成本。

#### 开发项

2. 增加开发规范文档
3. 增加迁移规范说明

5. 增加故障排查指引

#### 建议新增文档
- `docs/开发规范.md`
- `docs/迁移规范.md`
- `docs/运维与排障.md`

#### 验收标准
- 新开发者能按文档启动项目
- 能理解迁移、测试、日志和工作台能力

### 3. 推荐开发顺序

建议顺序如下：

1. CI 与迁移守护
作用：先把“持续开发不出事故”的底座打稳。

2. Observability
作用：出问题时能快速知道为什么。

3. Workspace Dashboard
作用：把现有能力整合成更易消费的接口。

4. Notification 抽象
作用：让系统从被动查询变成主动提醒。

5. 文档补齐
作用：降低维护成本和交接成本。

### 4. 具体任务拆分

#### 4.1 CI 与迁移守护任务拆分

1. 新建 GitHub Actions workflow
2. 配置 Python 环境和依赖安装
3. 执行 `pytest -q`
4. 执行迁移 smoke 流程
5. 输出测试与迁移结果
6. 更新 README

#### 4.2 Observability 任务拆分

1. 新增 request id 中间件
2. 给日志上下文注入 request id
3. 增加慢请求日志阈值配置
4. 增加异常结构化日志
5. 扩展 `/health` 输出模型
6. 补测试

#### 4.3 Dashboard 任务拆分

1. 新增 schema
2. 新增 `WorkspaceService`
3. 聚合现有 today/overdue/blocked/ready/recently-updated
4. 增加 REST 路由
5. 增加测试
6. 更新 README

#### 4.4 Notification 任务拆分

1. 定义 provider interface
2. 实现 webhook provider
3. 新增 alert dispatch service
4. 新增 dispatch endpoint
5. 增加发送日志
6. 可选记录发送历史

### 5. 风险与注意事项

#### 风险 1
Observability 做太重，反而引入复杂度。  
处理：先做最小闭环，不急着接 Prometheus / OpenTelemetry。

#### 风险 2
Notification 一开始耦合某个外部渠道。  
处理：先抽象 provider，再接具体实现。

#### 风险 3
Dashboard 逻辑重复实现，导致和现有视图不一致。  
处理：只做编排，不复制查询规则。

#### 风险 4
CI 依赖外部环境太多。  
处理：先保证 SQLite 路线稳定，PostgreSQL 可后续补独立 job。

### 6. 完成标准

当以下条件满足时，可以认为下一阶段完成：

1. PR 自动触发测试和迁移检查
2. 请求日志和异常日志可追踪
3. `/health` 能反映更完整系统状态
4. 存在统一 `workspace/dashboard` 接口
5. 存在主动提醒发送能力
6. 文档完整覆盖开发、迁移、运维说明

### 7. 对当前项目的意义

这阶段的意义不是“再加几个接口”，而是把系统从“功能可用”推进到“团队可长期维护”。

通俗说：
- CI 是防止以后改坏。
- 观测是出了问题能定位。
- Dashboard 是让系统更好用。
- 通知是让系统更主动。
- 文档是让后面接手的人不痛苦。

如果需要，下一步可以直接按这份文档开始推进第一项：CI 与迁移守护自动化。
