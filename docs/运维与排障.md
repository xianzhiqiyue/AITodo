# 运维与排障

## 健康检查

访问 `/health` 可查看：

- `status`
- `database`
- `migration`
- `parsing_service`
- `embedding_service`
- `version`

## 请求追踪

所有 HTTP 响应都会返回 `X-Request-ID`。

排查问题时，优先用这个值去查日志。

## 常见问题

### 1. `/health` 中 `migration = missing`

说明数据库已连接，但尚未执行 Alembic 迁移。

处理：

```bash
python -m alembic upgrade head
```

### 2. 通知分发失败

检查：

- `NOTIFICATION_WEBHOOK_URL` 是否已配置
- 目标 webhook 是否可访问
- 日志里是否出现 `notification_delivery_failed`

### 3. 请求变慢

超过 `SLOW_REQUEST_THRESHOLD_MS` 的请求会写 `slow_request` 日志。

优先排查：

- 数据库连接状态
- 最近新增的聚合查询
- 外部模型或 webhook 调用

### 4. 本地初始化失败

优先执行：

```bash
pip install -r requirements.txt
pytest -q tests/test_migrations.py
python -m alembic upgrade head
```

### 5.5 回滚到 database 模式

如果 native 模式出现问题，优先回滚模式，不回滚数据库迁移。

编辑：

```bash
/opt/ai-todo-server/docker-compose.yml
```

将：

```env
AITODO_STORAGE_MODE=obsidian_native
```

改为：

```env
AITODO_STORAGE_MODE=database
```

重启：

```bash
cd /opt/ai-todo-server
docker compose up -d ai-todo-api
```

验证：

```bash
curl -fsS http://127.0.0.1:8000/health
```

说明：迁移 `005` / `006` 只新增表，database 模式可继续使用原 `tasks` 主表。

### 5.7 常见问题

#### AITodo 访问 obsidianSync 失败

如果日志出现：

```text
httpx.ConnectError: All connection attempts failed
```

检查 `OBSIDIAN_SYNC_BASE_URL` 是否错误。

在容器内不要使用：

```text
http://127.0.0.1:3000/api/v1
```

应使用 Docker 网关，例如：

```text
http://172.21.0.1:3000/api/v1
```

#### obsidianSync 登录提示 refresh token 唯一键冲突

已修复：token payload 加入随机 `jti`。如再次出现，确认 obsidianSync `sync-api` 已部署最新版本。
