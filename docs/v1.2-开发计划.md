# v1.2 开发计划

## 1. 开发目标

完成“任务规划与主动协作阶段”的核心能力，让系统从任务中枢演进为更主动的执行助手。

## 2. 里程碑

### 里程碑 1：今日执行建议

开发项：

1. 设计 `suggested_today` schema
2. 新增 `ExecutionSuggestionService`
3. 增加 REST 与 MCP 接口
4. 补 service / API 测试

验收标准：

- 能返回建议任务
- 每条建议带原因说明
- 能按 `top_n`、标签过滤

### 里程碑 2：阻塞恢复建议

开发项：

1. 新增 `BlockedRecoveryService`
2. 实现阻塞原因摘要
3. 实现恢复建议输出
4. 增加 REST 与 MCP 接口

验收标准：

- blocked 任务可输出恢复建议
- 返回结果具备可解释性

### 里程碑 3：通知渠道扩展

开发项：

1. 在现有 provider 接口基础上新增真实渠道 provider
2. 增加通知测试接口
3. 补 provider 配置和文档

验收标准：

- 除 webhook 外至少支持一个真实渠道
- 能测试渠道配置

### 里程碑 4：完整任务计划生成

开发项：

1. 设计 `task plan` schema
2. 增强 `TaskPlanningService` 或新增 `TaskPlanService`
3. 支持计划生成与确认落库
4. 支持避免重复创建已有项

验收标准：

- 能输出结构化计划
- 能确认后批量创建任务与依赖

### 里程碑 5：任务回顾与总结

开发项：

1. 新增 `ReviewSummaryService`
2. 汇总阶段性任务指标
3. 输出最近进展摘要
4. 增加 REST 与 MCP 接口

验收标准：

- 支持按时间范围返回总结
- 支持基本过滤条件

## 3. 推荐顺序

1. 今日执行建议
2. 阻塞恢复建议
3. 通知渠道扩展
4. 完整任务计划生成
5. 任务回顾与总结

## 4. 测试要求

每个里程碑都至少补：

1. service 层测试
2. API 层测试
3. 必要时 MCP smoke test

提交前应执行：

```bash
pytest -q
```

## 5. 风险

1. 规则过多导致建议不稳定
2. 通知渠道接入带来外部依赖波动
3. 计划生成若过度依赖启发式，可能效果一般

## 6. 风险处理

1. 优先做最小闭环，再逐步增强排序和建议质量
2. 渠道接入采用 provider 抽象
3. 推荐结果始终保留可解释字段
