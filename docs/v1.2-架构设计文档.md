# v1.2 架构设计文档

## 1. 总体方向

v1.2 以现有单体后端为基础，继续保持单服务部署，但在内部增加更清晰的“规划、建议、通知、总结”编排层。

总体原则：

- 不拆微服务。
- 保持 REST 与 MCP 双入口。
- 新能力优先通过独立 service 组织，而不是继续堆到 `TaskService`。

## 2. 分层结构

### 2.1 接口层

- FastAPI REST routes
- MCP tools
- 请求校验、认证、响应模型

### 2.2 应用编排层

- `WorkspaceService`
- `AlertDeliveryService`
- `TaskPlanningService`
- 新增：
  - `ExecutionSuggestionService`
  - `BlockedRecoveryService`
  - `ReviewSummaryService`
  - 可选 `TaskPlanService`

### 2.3 领域服务层

- `TaskService`
- `TaskParsingService`

领域服务负责底层规则：

- 状态机
- 层级规则
- 依赖规则
- 搜索与视图查询

### 2.4 持久化层

- SQLAlchemy models
- Alembic migrations
- PostgreSQL / SQLite 双路径

### 2.5 外部适配层

- Embedding provider
- Parsing provider
- Notification providers

## 3. 新增模块关系

### 3.1 ExecutionSuggestionService

输入：
- `ready_to_start`
- 逾期信息
- 最近更新时间
- 父任务 / 依赖上下文

输出：
- 建议优先处理的任务列表
- 每个任务的推荐原因

### 3.2 BlockedRecoveryService

输入：
- blocked 任务
- 评论 / 时间线
- 依赖关系

输出：
- 阻塞原因摘要
- 恢复建议

### 3.3 ReviewSummaryService

输入：
- 指定时间范围内的任务、时间线、状态变化

输出：
- 汇总统计
- 最近进展
- 风险任务摘要

### 3.4 NotificationProvider 扩展

沿用当前 provider 抽象，新增：
- `DingTalkNotificationProvider` 或 `EmailNotificationProvider`
- `test` 能力用于验证通道配置

## 4. 核心数据流

### 4.1 任务计划生成流

1. 用户请求对任务生成计划
2. `TaskPlanService` 或增强后的 `TaskPlanningService` 读取任务上下文
3. 输出结构化计划
4. 用户确认后批量创建子任务和依赖

### 4.2 今日建议生成流

1. 请求 `suggested_today`
2. `ExecutionSuggestionService` 拉取 ready、overdue、recently_updated 等数据
3. 计算排序分数和原因
4. 返回建议列表

### 4.3 阻塞恢复流

1. 请求 blocked 任务恢复建议
2. `BlockedRecoveryService` 读取任务详情、依赖、评论时间线
3. 生成恢复建议
4. 返回结果给用户或 Agent

### 4.4 复盘总结流

1. 请求时间范围总结
2. `ReviewSummaryService` 汇总任务和事件
3. 输出统计和摘要

## 5. 数据模型建议

### 5.1 可选新增表

#### `task_plans`

- `id`
- `task_id`
- `source`
- `status`
- `meta_data`
- `created_at`

#### `task_plan_items`

- `id`
- `plan_id`
- `title`
- `description`
- `priority`
- `order`
- `depends_on_item_ids`
- `rationale`

### 5.2 可先不落库的能力

以下能力可先以 response-only 结构实现：

- `suggested_today`
- `recovery_suggestions`
- `review_summary`

## 6. 设计约束

1. 继续保持 SQLite fallback。
2. 所有新增 API 需要补 MCP 对应能力。
3. 所有推荐结果必须有可解释字段。
4. 不引入额外消息队列作为前置依赖。

## 7. 推荐实现顺序

1. `ExecutionSuggestionService`
2. `BlockedRecoveryService`
3. 新通知 provider
4. `TaskPlanService`
5. `ReviewSummaryService`
