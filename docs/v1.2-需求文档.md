# v1.2 需求文档

## 1. 背景

当前系统已经具备任务中枢的主干能力：

- 任务 CRUD、层级、状态机、依赖关系
- 自然语言解析与确认入库
- 工作台视图与 dashboard
- 评论 / 时间线 / 系统事件
- 提醒扫描与 webhook 通知
- AI 拆解建议与自动依赖落库
- CI、迁移守护、基础观测

下一阶段不再解决“能不能用”，而是解决“能不能更主动、更智能地推进任务执行”。

## 2. 阶段名称

`v1.2 任务规划与主动协作阶段`

## 3. 阶段目标

1. 让系统更会安排任务，而不只是存任务。
2. 让系统更主动推动执行，而不只是被查询。
3. 让任务数据更适合 AI 做计划、决策和回顾。
4. 让通知、工作台、规划建议形成闭环。

## 4. 目标用户

1. 用 AI 协助推进项目的个人开发者。
2. 小团队任务协作场景。
3. 需要让 Agent 根据任务上下文持续行动的系统。

## 5. 核心场景

1. 用户输入一个较大的目标，系统不仅能拆解，还能给出执行顺序和依赖。
2. 用户每天打开系统时，能直接看到“今天建议先做什么”。
3. 一个任务长期卡住时，系统会主动提醒，并给出可能的下一步动作。
4. Agent 执行完任务后，能回写结果并推动后续任务进入可执行状态。
5. 用户能查看某个目标从创建、拆解、执行到完成的完整过程。

## 6. 核心需求

### 6.1 任务规划增强

目标：从“建议拆解”升级到“可执行计划生成”。

需求：
1. 支持为一个任务生成结构化执行计划。
2. 计划至少包含建议子任务、推荐顺序、推荐依赖、每一步 rationale、风险点 / 前置条件。
3. 支持“只生成计划，不落库”。
4. 支持“确认计划后批量创建任务和依赖”。
5. 支持对现有任务重新规划，避免重复创建已有子任务。

### 6.2 今日执行建议

目标：明确回答“现在该做什么”。

需求：
1. 新增“今日建议任务”能力。
2. 在现有 `ready_to_start` 基础上综合优先级、截止时间、上层依赖、停滞时间、重点目标进行排序。
3. 返回每条建议任务的原因说明。
4. 支持 `top_n`、标签过滤和项目过滤。

### 6.3 阻塞恢复建议

目标：让 blocked 任务可被推动恢复。

需求：
1. 输出 blocked 任务的阻塞原因摘要。
2. 支持为 blocked 任务生成恢复建议。
3. 恢复建议至少包括联系人 / 前置输入 / 建议补充信息 / 是否可拆分继续推进。
4. 支持手动触发。

### 6.4 通知渠道扩展

目标：让主动提醒更接近真实使用场景。

需求：
1. 保留 webhook provider。
2. 新增至少一个真实渠道 provider，优先建议钉钉机器人或邮件。
3. 复用统一 provider 接口。
4. 支持按 `overdue` / `due_soon` / `blocked` 过滤发送。
5. 支持消息模板。
6. 发送失败和成功都要可追踪。

### 6.5 工作台进一步产品化

目标：从数据聚合升级为任务首页。

需求：
1. 保留现有 dashboard。
2. 新增以下板块之一或全部：
   - `suggested_today`
   - `stale_tasks`
   - `blocked_needing_attention`
   - `recent_failures`
3. 支持摘要计数。
4. 支持排序说明。
5. 支持前端或 Agent 一次性获取完整上下文。

### 6.6 任务回顾与总结

目标：支持阶段性复盘。

需求：
1. 支持按时间范围汇总任务进展。
2. 支持按标签、项目、父任务筛选。
3. 输出内容至少包括新建任务数、完成任务数、逾期任务数、长期阻塞任务数、最近主要进展。
4. 后续可扩展 AI 自动总结。

## 7. 非功能需求

### 7.1 稳定性

- 所有新增功能必须保持现有测试全绿。
- 新增模块必须有服务层测试和 API 测试。
- 迁移必须支持 SQLite / PostgreSQL 两条路径。

### 7.2 可维护性

- 新增规划、建议、总结逻辑尽量进入独立 service。
- 不把编排逻辑继续堆进 `TaskService`。
- REST 与 MCP 尽量复用相同 service。

### 7.3 可解释性

- 所有推荐结果尽量附带 `reason` / `rationale`。
- 不只返回排序后的任务列表。

### 7.4 可扩展性

- 通知 provider 必须可插拔。
- 规划和总结能力后续可切换到 LLM 或混合策略。

## 8. 接口建议

建议新增：

1. `POST /api/v1/tasks/{task_id}/plan`
2. `POST /api/v1/tasks/{task_id}/apply-plan`
3. `GET /api/v1/workspace/suggested-today`
4. `GET /api/v1/tasks/{task_id}/recovery-suggestions`
5. `GET /api/v1/workspace/stale`
6. `GET /api/v1/reviews/summary`
7. `POST /api/v1/notifications/test`

## 9. MCP 能力建议

建议同步新增：

- `plan_task_execution`
- `apply_task_plan`
- `get_suggested_today_tasks`
- `get_task_recovery_suggestions`
- `get_review_summary`
- `test_notification_channel`

## 10. 优先级

### P0

- 今日执行建议
- 通知渠道扩展
- 阻塞恢复建议

### P1

- 完整任务计划生成与确认落库
- 工作台增强

### P2

- 任务回顾与总结

## 11. 验收标准

1. 系统能生成更完整的执行计划，而不只是简单拆解。
2. 系统能明确输出“今天建议先做什么”。
3. blocked 任务能得到恢复建议。
4. 至少支持一个 webhook 之外的真实通知渠道。
5. dashboard 能输出更接近首页视图的聚合结果。
6. 新增能力均有测试覆盖，并保持全量测试通过。
